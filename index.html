Human-in-the-Loop AI Engineering

Mapping the V-model's phases to specialized AI agents, with humans as the final gate.

More details

The V-model maps development phases to their corresponding verification and validation steps. In an AI-driven SDLC, specialized agents can be assigned to each phase.

  • Requirements Agent: extracts, refines, and structures stakeholder needs.
  • Design Agent: translates requirements into system and software architecture.
  • Implementation Agents: generate and review code.
  • Test Agent: defines and executes verification and validation activities.

Key Principles

  • Test-driven development (TDD): tests are defined early and guide implementation throughout the lifecycle.
  • Agent independence: reduces shared context to avoid correlated or common-mode errors.
  • Small context scope: keeps each agent focused and improves reliability and consistency.
  • Cost and token optimization: limiting context per agent reduces token usage and improves scalability, runtime efficiency, and compute cost.
  • Clear separation of concerns: aligns with the structured verification logic of the V-model.
  • HUMIL (Human-in-the-Loop): humans remain the final safety and quality gate, ensuring oversight, domain judgment, and accountability across the lifecycle.
  • Capability-based tailoring: skills and roles can be used to adapt content and outputs to the task.

Overall, this enables a modular, traceable, cost-efficient, and more robust AI-supported SDLC.

Code Documentation Matters

AI output is only as good as the context it gets — and documentation is that context.

More details

AI coding assistants such as Claude Code and Copilot are becoming part of daily software engineering, but their output is only as good as the context they receive. In practice, prompt engineering increasingly becomes context engineering.

Well-documented projects help both humans and AI understand not only the code itself, but also the reasoning behind it.

  • Clean code and clear naming make software easier to understand for humans and AI.
  • Useful comments should explain why, not only what.
  • Architecture documentation clarifies system structure, interfaces, and dependencies.
  • Architecture Decision Records (ADRs) capture why important technical decisions were made.
  • Good code documentation is one of the quickest and easiest wins when working with AI tools.

What Better Documentation Enables

Without documentation, AI mainly sees implementation details. With documentation, AI can also understand design constraints, trade-offs, architecture decisions, business logic, and domain-specific topics.

  • This improves code suggestions, refactoring quality, onboarding speed, and the reliability of AI responses.
  • Better documentation can also reduce token usage and therefore lower cost, because concise documentation often provides context more efficiently than large amounts of raw source code.
  • From personal experience, this is one of the easiest practical wins because AI is very fast at routine documentation tasks.
  • In fast-growing code bases, software quality matters for humans and AI alike.
  • A practical approach is to keep a Markdown file that defines how and what should be documented, and update it after every AI-assisted code change together with the production code documentation.
  • For human understanding, block diagrams are especially useful; Mermaid is a practical tool for creating them.
